# 권한/기능 정책 기준서 (Super Admin · 조직관리자 · 사용자 · 발견자)

본 문서는 Pet-ID Connect의 **기준 권한 정책(RBAC + 운영 정책)** 을 고정 기록한 문서입니다.  
향후 권한 변경/기능 추가 시 이 문서를 먼저 확인하고, 코드/DB/화면 정책을 동일하게 맞춥니다.

---

## 1) 역할 정의

- **슈퍼관리자 (Super Admin)**: 플랫폼 전체 운영 권한을 가진 관리자
- **조직관리자 (Tenant Admin/Owner)**: 본인 소속 1개 조직만 운영하는 관리자
- **사용자 (Guardian/User)**: 일반 로그인 사용자(개인 대시보드/스토어/주문 이용)
- **발견자 (Finder/Public Viewer)**: 로그인 없이 공개 프로필/검증 화면을 보는 사용자

---

## 2) 역할별 권한 매트릭스

## 슈퍼관리자
- 조직 생성/수정/중지/멤버 권한 부여 가능
- `owner/admin/member` 모든 역할 부여 및 변경 가능
- 전체 조직/전체 사용자/전체 태그/전체 주문 조회 가능
- 슈퍼어드민 대시보드 및 플랫폼 운영 메뉴 접근 가능

## 조직관리자
- 본인 소속 조직의 멤버/태그/운영 데이터만 조회·관리 가능
- 조직 멤버 추가 자체는 가능하나, **관리자(owner/admin) 권한 부여는 불가** (member만 부여)
- 다른 조직 데이터 접근 불가
- 조직관리자는 운영 정책상 **1인 1조직 관리자 소속**만 허용

## 사용자
- 모든 모드 대시보드 **접속(조회)** 가능
- 본인 등록 모드는 쓰기 기능(등록/수정/태그연결/보증서 발급 등) 가능
- 미등록 모드는 읽기 전용(쓰기 기능 잠금)
- 스토어/주문은 로그인 사용자라면 사용 가능, **타 모드 상품 구매도 가능**

## 발견자
- 로그인 없이 공개 URL(프로필/검증) 접근 가능
- 개인정보/관리 기능 접근 불가
- 수정/삭제/연결/관리 액션 불가

---

## 3) 핵심 운영 규칙 (반드시 유지)

- 조직 생성은 **슈퍼관리자만** 가능
- 조직관리자(owner/admin) 권한 부여/초대는 **슈퍼관리자만** 가능
- 조직관리자 계정은 **복수 조직 관리자 소속 금지**
- 사용자 관점에서 조직(대리점) 개념은 최소 노출, 필요 화면만 노출
- 조직관리 진입은 조직 선택보다 **자동 진입(단일 조직 기준)** 우선
- 예외 상태 처리
  - 관리자 소속 0개: 권한 없음 안내 + 슈퍼관리자 문의
  - 관리자 소속 2개 이상: 정책 위반 안내 + 슈퍼관리자 정리 요청

---

## 4) 권한 변경 시 필수 체크리스트

- 정책 변경 이유/범위 문서화 (이 파일 갱신)
- 서버 권한 체크 코드 반영
  - 권한 검증 함수(예: admin authz, mode feature guard) 우선 수정
  - 우회 가능한 직접 액션/API 없는지 확인
- UI 노출 정책 반영
  - 메뉴 노출/비노출
  - 버튼 disabled + 안내 문구 일치
- 데이터 스코프 검증
  - tenant 단위 조회 필터 누락 여부 확인
  - 슈퍼관리자와 조직관리자 조회 범위 분리 확인
- 테스트/검증
  - 역할별 시나리오(슈퍼관리자/조직관리자/사용자/발견자) 수동 테스트
  - 타입체크/린트 통과 확인

## 5) 현재 정책의 주요 코드 기준점

- 공통 관리자 권한 계층: `src/lib/admin-authz.ts`
- 테넌트/조직 관리 액션: `src/app/actions/admin-tenants.ts`
- 사용자/모드 기능 가드: `src/lib/mode-visibility.ts`
- 허브/조직 진입 UI: `src/app/hub/page.tsx`
- 조직 관리 자동 진입: `src/app/hub/org/manage/page.tsx`
- 조직 생성 진입/권한 가드: `src/app/hub/org/new/page.tsx`, `src/app/actions/tenancy.ts`
